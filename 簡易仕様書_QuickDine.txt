# 簡易仕様書 — QuickDine

### 作者
Vrowdice

### アプリ名
QuickDine

#### コンセプト
地図上で検索位置を選び、日本国内の飲食店をすぐ探せる。

#### こだわったポイント
- 全画面 Google マップを中心にした検索 UX（地図タップ・GPS・クイックピン）
- 検索結果を地図マーカーとボトムシートの両方で表示
- お気に入り・クイックピンのローカル保存
- 韓国語 / 日本語 / 英語の多言語対応
- ホットペッパーグルメ Web サービスのクレジット表記を利用規約どおり実装

| 保存キー | 内容 |
|---------|------|

### 開発環境構築手順(ウェブアプリ)
1. リポジトリをクローン
2. `flutter pub get` を実行
3. `assets/env.example` を `assets/env` にコピーし、API キーを設定
   - HOTPEPPER_API_KEY
   - GOOGLE_MAPS_API_KEY
4. Android: `assets/env` の Maps キーは Gradle ビルド時に AndroidManifest へ注入
5. iOS: 起動時に `MapsKeyService` が env の Maps キーをネイティブへ渡す
6. `flutter run` で実機またはエミュレーター起動
7. HotPepper は日本国内データのみ → 海外からテストする場合は日本の座標で検索

## アプリケーション機能

### 機能一覧
- 飲食店検索：ホットペッパーグルメサーチ API を使用し、指定座標・検索半径（300m〜3000m）・取得件数（最大100件）で周辺店舗を検索する
- 地図検索：Google Maps 上でタップして検索中心を指定する
- 現在地取得：GPS で現在地を取得し、地図中心を移動する
- クイックピン：よく使う地点に名前付きピンを保存し、ワンタップで移動する
- 検索結果表示：地図上に店舗マーカーを表示し、ボトムシートで一覧表示する
- 店舗詳細表示：店名・写真・住所・営業時間・アクセス情報を表示する
- お気に入り：店舗を端末内に保存・解除し、一覧から再表示できる
- 地図連携（アプリ内）：詳細画面またはお気に入りから店舗位置を検索地図上に表示する
- 設定：デフォルト検索半径・取得件数・表示言語の変更、お気に入り・クイックピンの一括削除
- クレジット表示：API データ・画像利用画面にホットペッパー公式クレジットを表示する

### 画面一覧
- 検索画面（SearchScreen）：全画面地図、検索半径・取得件数選択、検索実行、結果シート、クイックピンパネル、お気に入り・設定への導線
- 店舗詳細画面（DetailScreen）：店舗情報の詳細表示、お気に入り登録、地図表示への戻り
- お気に入り画面（FavoritesScreen）：保存済み店舗の一覧
- 設定画面（SettingsScreen）：検索デフォルト値、言語、データ削除、アプリ情報

### 使用しているAPI,SDK,ライブラリなど
- ホットペッパーグルメサーチ API（Recruit Web Service）
- Google Maps SDK for Android / iOS（google_maps_flutter）
- Flutter パッケージ
  - http（API 通信）
  - geolocator（GPS・位置権限）
  - flutter_dotenv（API キー読み込み）
  - shared_preferences（ローカル保存）
  - url_launcher（クレジットリンク起動）
  - flutter_localizations / intl（多言語）

## 開発における設計判断と工夫

### 1. 技術スタック選定と生成型 AI の活用について

#### 開発言語の選定
課題ガイドラインで許可されている言語のうち、Kotlin・Swift の経験がない点を踏まえ、すでに習熟している **Flutter / Dart** を選択した。短期間で要件を安定して実装し、「コードを十分理解しているか」という評価基準にも応えるための判断である。

#### AI 活用の責任
課題では生成型 AI の使用が許可されている一方、「生成されたコードを十分に理解せず使用したと判断された場合、評価に影響する可能性がある」と明記されている。これに従い、Cursor 等が生成したボトムシートの状態管理や API 呼び出しロジックなどについて、コメントを丁寧に付け、構造を把握するプロセスを経てから採用した。

---

### 2. セキュリティと提出方式について

#### API キー管理
パブリック GitHub へソースコードを提出する際、ホットペッパー API キーの漏洩を防ぐため `assets/env` を `.gitignore` で除外した。

#### 採点者への配慮
代わりに `assets/env.example` をリポジトリに残し、本仕様書の「開発環境構築手順」に、採点者が自身の API キーを記入してテストできる手順を記載した。

### 3. 課題必須要件と UX のエンジニアリング上のジレンマ

課題の必須要件を満たしつつ、使い勝手を損なわないための折衷点を以下のように設計した。

#### 検索半径 UI vs データ過負荷の防御

| 項目 | 内容 |
|------|------|
| **課題** | 「現在地からの検索半径を指定できること」が必須要件 |
| **懸念** | 東京駅など都心で 3000m 半径を検索すると、大量の店舗データが返り、地図描画の低下や API 負荷が懸念された |
| **解決** | 5 段階（300m〜3000m）の半径選択 UI を実装して要件を満たす。加えて、課題のページング要件（「最初のページのみ表示でも可」）を応用し、1 回の API リクエストで取得する **最大件数（max count: 10〜100 件）** をユーザーが制限できるようにした |

#### 必須「一覧画面」 vs 地図中心の探索フロー

| 項目 | 内容 |
|------|------|
| **課題** | 検索結果を店名・アクセス情報・サムネイル付きの「一覧」で表示することが必須 |
| **懸念** | 一覧を別画面に分離すると、地図中心の探索フローが途切れる |
| **解決** | 全画面地図を維持したまま、画面下部から引き上げられる **DraggableScrollableSheet（ボトムシート）** を導入。地図マーカービューとリストビューを融合し、要件を満たしつつ UX を維持した |

#### 詳細画面から戻った際の検索状態の保持

| 項目 | 内容 |
|------|------|
| **課題** | 詳細画面で「地図で見る」を押して戻ると、カメラ移動（moveTo）に伴い、既存の検索結果マーカー（オレンジ色）が消えてしまう |
| **解決** | 不要な `moveTo()` 呼び出しを避け、座標のみ更新するよう状態管理を調整。`_searchResults` が失われず、検索結果が地図上に維持されるようにした |

### 4. UI/UX のディテールと使いやすさの改善

機能が動作するだけでなく、実サービス水準の滑らかな体験を目指し、以下の調整を行った。

- **UI 重なり（Overlap）の解消**
  上部に検索コントロールを追加した結果、クイックピン・現在地（GPS）ボタンが隠れる問題を解消。現在地ボタンを右下へ移動し、クイックピン側には上部余白を確保して再配置した。

- **片手操作の最適化**
  届きにくい右上の虫眼鏡アイコンに代わり、画面下部中央に **Pill 型の大きなフローティング検索ボタン** を配置し、タップしやすさを向上させた。

- **アニメーションフィードバック**
  クイックピンボタン押下時にスケール／バウンスアニメーションを追加。ボトムシートの表示・非表示にも滑らかなアニメーションを適用し、操作の手応えと完成度を高めた。
